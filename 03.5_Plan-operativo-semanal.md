### Mecanismos de Redundancia y Calidad en Pares

| Mecanismo | Implementación | Responsable Validación |
|-----------|----------------|------------------------|
| **Implementación en pares** | Un alumno codifica/configura; el otro observa y documenta | Rotación diaria de roles dentro del par |
| **Validación cruzada entre pares** | Pares de un stream validan entregables de otro stream (ej. BD valida ADD de ERP) | Arquitectos (Alumnos 1-4) coordinan validaciones |
| **Revisión por pares en GitHub** | Todo *pull request* requiere aprobación de al menos 1 compañero fuera del par original | Prof. Zavala revisa *pull requests* críticos antes de fusión |
| **Ceremonias de calidad** | Miércoles laboratorio: Demostración funcional de entregables ante el grupo completo | Prof. Zavala como facilitador y árbitro final |
| **Documentación obligatoria** | Cada actividad genera diagrama UML o checklist validado por el par | Business Analysts (Alumnos 27-30) auditan completitud |

> "Construir una cultura empoderada requiere crear un entorno donde los miembros del equipo sean confiados, respetados y dotados de autonomía para tomar decisiones dentro de sus áreas de especialización" (PMI, 2025, p. 53, traducción propia).

### Flujo de Trabajo GitHub con Control Estricto del Profesor Zavala

```bash
# Estructura de ramas (controlada por Prof. Zavala)
main          # Producción simulada - SOLO Prof. Zavala fusiona aquí
├── release/1.1.0  # Liberación Semana 1 - SOLO Prof. Zavala crea/mergea
├── release/1.2.0  # Liberación Semana 2 - SOLO Prof. Zavala crea/mergea
└── develop        # Integración TEST - Arquitectos aprueban PRs hacia aquí
    ├── feature/rocky-lvm        # Pares 5+6
    ├── feature/freeipa-ldap     # Pares 7+8
    └── feature/postgres-schema  # Pares 9+10

# Proceso semanal (miércoles 19:00 cierre)
1. Alumnos crean PR desde feature → develop
2. Arquitectos (Alumnos 1-4) revisan y aprueban PRs
3. Prof. Zavala revisa develop → crea rama release/1.X.0
4. Prof. Zavala valida release → crea tag v1.X.0
5. Prof. Zavala fusiona release → main y publica release oficial
```

*El Profesor Zavala mantiene permisos de administrador exclusivos en GitHub; los alumnos tienen permisos de escritura únicamente en ramas `feature/*`.*

### Checklist de Éxito del Proyecto

La PoC se considerará exitosa cuando se cumplan los siguientes criterios validados por el Profesor Zavala:

- [ ] **Integración funcional:** Los seis componentes operan en secuencia sin puntos únicos de falla (Nivel 0 → Nivel 4)
- [ ] **Configuración sin código:** Todos los procesos académicos modelados mediante Application Dictionary sin desarrollo Java personalizado
- [ ] **Cumplimiento normativo:** Gestión documental con trazabilidad completa conforme a LGPD mexicana; cifrado LUKS2 para datos sensibles
- [ ] **Transferencia de conocimiento:** Documentación UML completa como activo intangible institucional en el repositorio GitHub
- [ ] **Soberanía tecnológica validada:** Arquitectura 100% software libre sin dependencia de licenciamientos comerciales ni vendor lock-in
- [ ] **8 liberaciones semanales exitosas** en el repositorio https://github.com/jzavalar/SiE con etiquetas `v1.1.0` a `v1.8.0`
