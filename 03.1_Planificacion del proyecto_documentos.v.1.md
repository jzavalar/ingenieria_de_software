#### 1.1 Propósito y Justificación

El presente documento autoriza formalmente la ejecución de la Prueba de Concepto (PoC) del hipotético Sistema de Información Escolar (SiE) para la Unidad Iztapalapa de la Universidad Autónoma Metropolitana. La PoC validará técnicamente una arquitectura de software libre compuesta por seis componentes interdependientes (Niveles 0-4) para reemplazar sistemas legacy propietarios, garantizando soberanía tecnológica institucional y alineación con los procesos académicos específicos de la UAM.

> "El acta de proyecto es un documento emitido por el iniciador o patrocinador del proyecto que autoriza formalmente la existencia de un proyecto y proporciona al gestor del proyecto la autoridad para aplicar los recursos organizacionales a las actividades del proyecto" (PMI, 2025, p. 125, traducción propia).

#### 1.2 Objetivos del Proyecto

| Objetivo | Criterio de Éxito Medible |
|----------|---------------------------|
| Validar integración técnica secuencial de los seis componentes | Todos los componentes operativos en 8 semanas sin puntos únicos de falla |
| Demostrar configuración sin código mediante Application Dictionary | 100% de workflows académicos configurados sin desarrollo Java personalizado |
| Validar cumplimiento normativo LGPD mexicana | Implementación de mecanismos de cancelación, oposición y seguridad documental |
| Capitalizar conocimiento tácito como activo intangible | Documentación UML completa de arquitectura y configuraciones del ADD |
| Validar modelo de fábrica de software educativa | 30 alumnos con roles especializados entregando 2,640 horas-hombre en 8 semanas |

#### 1.3 Resumen de Alcance

**Entregables principales:**
- Entorno base Rocky Linux 9.x con SELinux enforcing y LUKS2 (Nivel 0)
- Directorio LDAP/FreeIPA funcional con esquema UAM (Nivel 1)
- Base de datos PostgreSQL 15 con esquema relacional académico (Nivel 2)
- Contenedor Tomcat 10 + Mayan EDMS 4.x con workflows documentales (Nivel 3)
- iDempiere 9.x configurado mediante Application Dictionary para procesos académicos UAM (Nivel 4)
- Documentación técnica UML como activo intangible institucional

**Exclusiones explícitas:**
- Migración de datos históricos de sistemas legacy
- Integración con sistemas de nómina institucional
- Despliegue en producción o entornos de usuario real
- Desarrollo de código Java personalizado fuera del Application Dictionary

#### 1.4 Hitos Principales

| Hito | Fecha Estimada | Criterio de Aceptación |
|------|----------------|------------------------|
| Hito 1: Entorno Rocky Linux operativo (Nivel 0) | Semana 3 | SELinux enforcing, particionamiento LVM validado, firewalld configurado |

| Hito 6: Cierre de PoC y demostración final | Semana 10 | 100% casos críticos aprobados, documentación UML completa |

#### 2.1 Enfoque de Gestión del Alcance

El alcance del proyecto se gestionará mediante un enfoque híbrido: predictivo para la secuencia de integración de componentes tecnológicos (Niveles 0-4) y adaptativo para la configuración iterativa de workflows académicos mediante el Application Dictionary de iDempiere. Este enfoque reconoce que:

> "En lugar de entregar todas las historias de usuario que se encuentran en el *backlog*, el equipo se enfoca en los objetivos principales del producto para satisfacer las expectativas del cliente. El alcance del proyecto se refina a través de diferentes iteraciones" (PMI, 2025, p. 45, traducción propia).

#### 3.1 Metodología de Desarrollo del Cronograma

El cronograma se desarrollará mediante planificación por oleadas progresivas (*rolling wave planning*), reconociendo la naturaleza híbrida del proyecto:

> "Una técnica bien conocida que ayuda con este enfoque es la planificación por oleadas progresivas (*rolling wave planning*)" (PMI, 2025, p. 48, traducción propia).

- **Semanas 1-2:** Plan detallado para Niveles 0 y 1 (Rocky Linux + FreeIPA)
- **Semanas 3-4:** Plan detallado para Nivel 2 (PostgreSQL) y primera mitad del Nivel 3 (Tomcat)
- **Semanas 5-6:** Plan detallado para segunda mitad del Nivel 3 (Mayan EDMS) e inicio del Nivel 4 (iDempiere base)
- **Semanas 7-8:** Plan detallado para workflows académicos (Nivel 4) y cierre

| **4** | Nivel 3a: Tomcat 10 | ✓ Contenedor operativo con memoria optimizada<br>✓ DataSource JDBC funcional hacia PostgreSQL<br>✓ SSO funcional para aplicaciones Java | Requiere Niveles 1 y 2 completos |
| **5** | Nivel 3b: Mayan EDMS | ✓ 5 tipos documentales configurados<br>✓ Workflows de aprobación operativos<br>✓ Mecanismos LGPD implementados | Requiere Niveles 1, 2 y 3a completos |

  - 0.3 semanas para integración Tomcat/Mayan (Semanas 4-5)
  - 0.3 semanas para configuración del Application Dictionary (Semanas 6-7)

#### 4.4 Métricas de Valor Ganado

| Métrica | Fórmula | Interpretación |
|---------|---------|----------------|
| Valor Planificado (PV) | % planificado × 2,640 h-h | Esfuerzo planificado acumulado |
| Valor Ganado (EV) | % completado × 2,640 h-h | Esfuerzo realmente entregado |
| Costo Real (AC) | Horas reales consumidas | Esfuerzo realmente invertido |
| Variación del Cronograma (SV) | EV – PV | SV > 0: Adelantado; SV < 0: Atrasado |
| Variación de Costos (CV) | EV – AC | CV > 0: Bajo presupuesto; CV < 0: Sobre presupuesto |
| Índice de Desempeño del Cronograma (SPI) | EV / PV | SPI > 1: Adelantado; SPI < 1: Atrasado |
| Índice de Desempeño de Costos (CPI) | EV / AC | CPI > 1: Bajo presupuesto; CPI < 1: Sobre presupuesto |
